## Topic 1: Misconception Resolution Triggers

### What signal reliably indicates a misconception is "cleared"?

**Source 1 — Option Tracing integrated into Knowledge Tracing (Journal of Learning Analytics, 2025)**
- URL: https://learning-analytics.info/index.php/JLA/article/view/8557
- Core finding: Modern KT models that track *which wrong option* a student chose (not just right/wrong) can detect when a student stops selecting the distracting misconception option on subsequent attempts. The model distinguishes between different incorrect choices, making "stopped choosing the misconception distractor" a stronger clearance signal than generic accuracy.
- Finding 2: Consecutive-error sequences in beginners do not auto-clear by subsequent correct answers alone — the 2024 data showed "consecutive errors in the same skill might lead to greater learning gains," implying that clearance requires a sustained shift, not a single correct answer.
- Source file: https://files.eric.ed.gov/fulltext/EJ1465691.pdf

**Source 2 — Mastery Learning threshold consensus (research.com, 2024 guide)**
- URL: https://research.com/education/what-is-mastery-learning
- Core finding: Most Mastery Learning frameworks define the clearance threshold at **≥80% accuracy** before a knowledge component is declared mastered. Higher thresholds (85–90%) are correlated with greater exam performance gains. This aligns with course-hub's existing 85% rule for difficulty adaptation.

**Source 3 — Deep Learning-based Knowledge Tracing Survey (Springer, 2024)**
- URL: https://link.springer.com/article/10.1007/s10489-024-05509-8
- Core finding: State-of-the-art KT models do not use a single "consecutive correct answers" rule for misconception clearance; instead they model a probabilistic learning state that must exceed a threshold (typically 0.8–0.9 posterior probability) across multiple attempts before a misconception KC is considered inactive.

**Actionable recommendation for course-hub:**

Auto-resolve a misconception when **all three** of the following hold:
1. The student answers the specific concept correctly on ≥3 of the last 4 attempts (75% local accuracy).
2. At least one of those correct answers occurred in a new session (not the same sitting — prevents lucky-streak gaming).
3. The misconception's `occurrence_count` has not incremented in the past 14 days.

This is more defensible than a single accuracy threshold. Implementation: add a `auto_resolved_at` nullable timestamp column to the `misconceptions` table; run a nightly Supabase Edge Function or check on quiz-submit to evaluate the three conditions and set `resolved = true` + `auto_resolved_at`.

**Estimated effort:** 1–2 days. Nightly Edge Function + DB column migration + UI badge on misconception list to show "auto-cleared."

## Topic 2: FSRS + Server-Side Storage Patterns

### Best practices for syncing FSRS state to a database while keeping client-side scheduling

**Source 1 — ts-fsrs official documentation and npm package**
- URL: https://open-spaced-repetition.github.io/ts-fsrs/
- URL (npm): https://www.npmjs.com/package/ts-fsrs
- Core finding: ts-fsrs exposes a `Card` interface with exactly 9 serializable fields: `due`, `stability`, `difficulty`, `elapsed_days`, `scheduled_days`, `reps`, `lapses`, `state`, `last_review`. Every field is a primitive (number, Date, enum integer). The `repeat()` function returns a `RecordLog` object containing both the updated card state and the review log entry for each possible rating — both must be persisted together atomically.
- Core finding 2: The library provides `rollback`, `forget`, and `reschedule` utilities that rebuild card state by replaying a stored review log. This means if you persist every `ReviewLog` row to Supabase, you can always reconstruct correct card state server-side, making the client localStorage a mere cache.

**Source 2 — awesome-fsrs curated list**
- URL: https://github.com/open-spaced-repetition/awesome-fsrs
- Core finding: Several production apps (Mochi Cards, RemNote, Logseq) implement an **offline-first / local-cache + server-of-record** pattern: the client computes the next review schedule locally using ts-fsrs, immediately updates the UI, then syncs the `Card` state + `ReviewLog` row to the server in the background. On next device login, the client hydrates from server state rather than localStorage.

**Source 3 — ts-fsrs Card interface fields (DeepWiki / CDN type definitions)**
- URL: https://deepwiki.com/open-spaced-repetition/ts-fsrs/4.2-card-creation-and-management
- URL (types): https://cdn.jsdelivr.net/npm/ts-fsrs@4.7.1/dist/index.d.ts
- Core finding: The `Card` object is fully JSON-serializable. Dates serialize to ISO strings; `State` and `Rating` are numeric enums. A single JSONB column in Postgres can hold the full card state, or you can normalize into columns (preferable for querying due dates).

**Recommended Supabase schema for course-hub FSRS migration:**

```sql
-- One row per (user, question) pair
CREATE TABLE fsrs_cards (
  id           uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id      uuid REFERENCES auth.users NOT NULL,
  question_id  uuid REFERENCES questions NOT NULL,
  -- FSRS Card fields (all from ts-fsrs Card interface)
  due              timestamptz NOT NULL DEFAULT now(),
  stability        float8 NOT NULL DEFAULT 0,
  difficulty       float8 NOT NULL DEFAULT 0,
  elapsed_days     int NOT NULL DEFAULT 0,
  scheduled_days   int NOT NULL DEFAULT 0,
  reps             int NOT NULL DEFAULT 0,
  lapses           int NOT NULL DEFAULT 0,
  state            smallint NOT NULL DEFAULT 0,  -- State enum: 0=New,1=Learning,2=Review,3=Relearning
  last_review      timestamptz,
  updated_at       timestamptz NOT NULL DEFAULT now(),
  UNIQUE (user_id, question_id)
);

-- Append-only review log for full replay capability
CREATE TABLE fsrs_review_logs (
  id           uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id      uuid REFERENCES auth.users NOT NULL,
  question_id  uuid REFERENCES questions NOT NULL,
  rating       smallint NOT NULL,   -- Rating enum: 1=Again,2=Hard,3=Good,4=Easy
  state        smallint NOT NULL,
  due          timestamptz NOT NULL,
  stability    float8 NOT NULL,
  difficulty   float8 NOT NULL,
  elapsed_days int NOT NULL,
  last_elapsed_days int NOT NULL,
  scheduled_days int NOT NULL,
  review       timestamptz NOT NULL DEFAULT now()
);
```

**Migration approach:** On app load, check if `fsrs_cards` has a row for the user. If yes, hydrate from DB (discard localStorage). If no, migrate the localStorage entries to DB on first authenticated load, then clear localStorage. Use `upsert` on `fsrs_cards` keyed on `(user_id, question_id)`.

**Estimated effort:** 2–3 days. DB migration + API route for upsert + hydration logic in quiz component + localStorage migration on first login.

## Topic 3: Optimal Wrong-Answer Feedback Timing

### Should specific error feedback come immediately or be delayed?

**Source 1 — "Timing of feedback and retrieval practice" (Nature Humanities and Social Sciences Communications, 2024)**
- URL: https://www.nature.com/articles/s41599-024-03983-6
- Core finding: Both immediate and delayed feedback groups significantly outperformed the no-feedback and restudying-only groups at 1-week and 1-month follow-up. Immediate feedback led to superior performance on immediate post-tests; delayed feedback showed marginally better transfer effects.

**Source 2 — "Timing's not everything" (Medical Education / Wiley, 2024)**
- URL: https://asmepublications.onlinelibrary.wiley.com/doi/full/10.1111/medu.15287
- Core finding: In formative multiple-choice testing contexts, post-item vs. post-block feedback timing had **no significant effect** on overall learning efficacy. The key driver was providing feedback at all, not its precise timing.

**Source 3 — Memory malleability and immediate corrective feedback (PMC)**
- URL: https://pmc.ncbi.nlm.nih.gov/articles/PMC9345471/
- Core finding: Corrective feedback is most effective at correcting false memories when delivered immediately after a retrieval attempt, when memory is in a malleable (reconsolidation) state. Delaying feedback allows the incorrect memory trace to restabilize, making it harder to overwrite.

**Source 4 — Scaffolded feedback for long-term error correction (Memory & Cognition / Springer)**
- URL: https://link.springer.com/article/10.3758/MC.38.7.951
- Core finding: "Scaffolded feedback" — where the student is first asked to self-generate the answer again before seeing the correct answer — produced the best delayed recall scores, outperforming both immediate-reveal and no-feedback conditions.

**Source 5 — Test-potentiated encoding / correct-answer feedback (Frontiers in Behavioral Neuroscience)**
- URL: https://www.frontiersin.org/journals/behavioral-neuroscience/articles/10.3389/fnbeh.2023.1100497/full
- Core finding: Providing the correct answer *after* a retrieval attempt (right or wrong) enhances encoding of that answer — called test-potentiated encoding (TPE). This effect is strongest when correct-answer feedback is explicit and complete, not just a "correct/incorrect" signal.

**Actionable recommendation for course-hub:**

Use **immediate feedback with full correct-answer explanation** for wrong answers. The evidence converges on three requirements:
1. Show feedback immediately after submit (not at end of quiz block) — memory is maximally malleable at this point.
2. Show the *correct answer explicitly* with explanation — "you answered X, correct answer is Y because Z." A bare "incorrect" signal does not produce TPE.
3. For high-value misconception questions, consider a one-step scaffolded prompt before revealing the answer: "Think again — what is the key principle here?" before auto-revealing. This adds ~10 seconds of generation effort that strengthens encoding.

Current course-hub behavior: check whether wrong-answer feedback currently shows the explanation or just marks it wrong. If only marking wrong, adding the explanation is a high-ROI, low-effort change.

**Estimated effort:** 0.5–1 day if explanation text is already stored per question (just render it in the feedback state). 2–3 days if explanation generation requires a new AI call per answer.
