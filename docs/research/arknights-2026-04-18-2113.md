## 关键发现（5 条对我们项目最有参考价值）

### 1. Targeting 的"三级骨架"——search cycle + 优先级栈 + 阻挡特权

**他们怎么做**（Terra Wiki）：
- 每个单位 **每 3 帧（0.1s）执行一次 search cycle**，与我们的 10Hz tick 完美对齐
- 优先级栈（从高到低）：`Blocked unit > Special Priority > Highest positive aggression > Closest to destination > Highest negative aggression`
- Ranged op 放在地面格并阻挡敌人时，**不会**优先打被自己挡住的敌人（反直觉，但官方如此）
- 敌人选目标用 `a = 1000 × (t + s)` 公式，s 是部署时间（上限 10000），近似"最晚部署者"

**48-Arknights 怎么做**：`CheckInRange()` 对 enemy list 做线性扫描，**第一个命中即返回**，不考虑距离/HP/优先级。

**我们现状**：`_blocked_enemy` 只返回第一个活敌，完全没实现优先级。

**是否应该抄**：**部分抄**。优先级栈全抄（尤其"被阻挡敌人优先"），但公式层面的 `1000(t+s)` 可简化为 enemy 的 `_path_progress` 大的优先（我们已经有这个字段）。**不要**抄 48-Arknights 的朴素实现，那会放弃 10Hz 架构优势。

### 2. Buff Pipeline 的两段式是标准

**他们怎么做**（Terra Wiki / GamePress）：
```
Atk_intermediate = FLOOR(Atk_base × Atk_stage) × (1 + Σ ratio_buff)
Atk_final        = FLOOR(Atk_intermediate × Π multiplier_buff × Π (1 - multiplier_debuff))
```
关键：**ratio 加法先算，multiplier 乘法后算，FLOOR 取整两次**。伤害 amplification（易伤/减伤）独立 multiplicative：`(1+x1)(1+x2)...`。DEF/RES 的 % 减同理。5% floor 在 "独立 multiplier" 之前应用。

**我们现状**：`_atk_bonus` 单加法字段，只能表达 ratio 总和。

**是否应该抄**：**应该抄**。改成 `Operator.atk_ratio_buffs: list[float]` + `atk_multiplier_buffs: list[float]` 两列表，伤害计算时做两阶段。这是 P4 SilverAsh 技能能正确 stack 其他 ATK buff 的前提。

### 3. AOE 是欧氏半径，不是切比雪夫

**他们怎么做**（Terra Wiki: Splash Caster）：
- Splash radius 普遍 **1.0–1.5 tiles**（浮点）
- 以目标位置为圆心的圆形半径，不是方形 grid
- Fiammetta 主弹 1.5、二次爆炸 1.1，都是真实圆形

**我们现状**：完全没做 AOE。

**是否应该抄**：**应该抄欧氏半径**。我们原 prompt 里提的 "circle（欧氏）/ square（切比雪夫）/ line 三种形状" 过度抽象——游戏里 99% 的 AOE 是 circle，只有少数（Siege S3 推拉、Aak S3 buff）才用矩形 range。建议 P6 先只实现 `splash_radius: float` 作为单元的属性，不做 shape 枚举。

### 4. SP 三模式 + lockout 语义

**他们怎么做**（Terra Wiki: Skill Point）：
- **Auto Recovery**: 1 SP / 秒 （独立于攻击）
- **Offensive Recovery**: 1 SP / 攻击 （治疗也算）
- **Defensive Recovery**: 1 SP / 被攻击 （dodge/resist 也算）
- **Lockout**：SP 满但没目标时进入"橙色"锁，回不了 SP；manual skill 满后永久锁定等玩家点击
- Overcharge：最多累积到 2 × SP cost

**我们现状**：只有 auto_time / auto_attack 两种回复模式，没 lockout、没 defensive、没 manual trigger。

**是否应该抄**：**大部分抄**。
- Defensive recovery 现在不用做，到有 Liskarm 类干员再加
- Lockout 必须做（否则 auto_skill 在没敌时会立刻再次触发，与游戏不符）
- Manual trigger 留给 P7，现在用户没真实 UI，意义不大
- Overcharge 暂不做（冷门）

### 5. 地形 elevation 在官方代码层是"tile.type 字段"，不是几何距离

**他们怎么做**（48-Arknights）：`Tile_Ground` (Y=-0.5) vs `Tile_Step` (Y=-0.3) 只是 3D 渲染高度，**没有**"高台 ranged op 可以打 ground 敌人"的显式代码——靠 range 格子的平面覆盖自然解决（ranged op 的 range 是 3×3 平面，敌人不管在什么 tile 上只要坐标在 range 内就能打）。

**我们现状**：`map.py` 定义了 tile 类型但没任何 range check 逻辑。

**是否应该抄**：**直接抄**。Elevation 不需要在 targeting 逻辑里额外判断——只要 range 用平面欧氏/曼哈顿距离覆盖，地面敌人自然进入高台 op 的射程。唯一例外：melee op 放在地面才能 block，放高台不能 block（这个规则已在我们 `attack_range: "melee"|"ranged"` 字段里隐含）。**P3 完成后可以宣布 elevation 已解决**，不用单独做 P6a。

## 优先级建议

**推荐：P3 完成后走 P6（AOE / Buff pipeline）而非 P5（CLI）或 P7（manual skill）**

**理由**（2 行）：
1. AOE 是单元类型的硬缺口——没它就做不了 Caster/Sniper 除狙击以外的所有分支，实际战斗简化到"物理单挑"失真严重；Buff pipeline 两段式是 SilverAsh + 任何团队 buff（Warfarin/Saileach）同时生效的前提，否则 P4 skill 系统是跛脚的
2. CLI（P5）和 manual skill（P7）本质是"外壳"，机制层跑通后随时可加；现在优先补机制让单测更有意义（15 tests → 可以扩到 30+ 覆盖 AOE + buff 组合）

**二级推荐**：P6 内做完 AOE + buff pipeline 后，顺手加 **targeting 策略栈**（发现 #1），因为 `_blocked_enemy` 返回第一个活敌的 hack 此时会成为 AOE 测试的 noise 源。

## 风险与局限

1. **Terra Wiki 是玩家逆向产物**：公式来自社区抓包 + 实验，非 HG 官方披露。`1000(t+s)` 这种常数可能在未来版本（如国服 2025Q2 大改）有偏差。建议在单测里不硬编码数值，写成"deploy 顺序决定优先级"的性质测试。
2. **48-Arknights 是 48h hack**：代码质量不具备工业强度（first-match targeting 就是证据），只能参考"如何把架构落成代码"，不能参考"它是否正确"。
3. **ASPD 30fps 冲突**：Terra Wiki 写"游戏 30 fps，攻击间隔被 round 到 frame"，我们 10Hz tick = 3 frame/tick。意味着 0.1s 以下粒度的 ASPD 差异我们表达不了（例如 1.0s vs 1.03s 间隔在我们这里等价）。现阶段可忽略，但 P5+ 做数值对比验证时会撞墙。
4. **没有找到 Python 机制模拟器参考**：意味着我们的设计是开创性的，但也没有"正确答案"可校验。建议：关键公式实现后，用游戏内已知场景（如 Liskarm tank Slug）的 wiki 战报数值做对拍。
5. **AOE 圆心在"目标"还是"投射物落点"**：Fiammetta 的 1.5 radius 是以击中点为圆心——我们因为没建模投射物飞行，只能近似为"以目标为圆心"。Flinger / 圆炮 远程弹道的滞留 AOE 现阶段做不到保真。
