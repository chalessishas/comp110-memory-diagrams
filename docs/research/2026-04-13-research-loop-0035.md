## Proposal 1: Homophone Confusion Detection (grammar.js) — HIGH VALUE

### Research Basis

Leacock et al. (2014) "Automated Grammatical Error Detection for Language Learners" (Synthesis Lectures on HLT): homophones are the #1 category of errors that bypass spell-checkers. Chinese L1 writers with TOEFL scores 20-25 (writing section) show ~15% homophone error rate in corpus data — higher than native writers' ~2%.

Current state: grammar.js relies on mechanics.js spell-checker which passes ALL homophones (they're/their/there all valid spellings). These errors signal L2 orthographic interference and directly reduce the ETS "language use" criterion score.

**High-confidence, low-FP patterns for Chinese L1:**

```js
// grammar.js — homophone confusion detection
// Leacock et al. 2014: homophones bypass spellcheck; Chinese L1 rate ~15% vs native ~2%
// Only patterns where context unambiguously reveals the error
const HOMOPHONE_ERRORS = [
  // "your" vs "you're" — context: followed by verb in present tense
  { re: /\byour\s+(is|are|was|were|have|has|had|do|does|did|can|could|will|would|should|must|going)\b/i,
    msg: 'Possible confusion: "your" (possessive) vs "you\'re" (you are)' },
  // "its" vs "it's" — "its" followed directly by a verb is almost always an error
  { re: /\bits\s+(is|are|was|were|have|has|been|going)\b/i,
    msg: 'Possible confusion: "its" (possessive) vs "it\'s" (it is/has)' },
  // "there" vs "their" vs "they're" — most unambiguous: "there think/believe/feel/want"
  { re: /\bthere\s+(think|believes?|feels?|wants?|needs?|hopes?|decides?)\b/i,
    msg: 'Possible confusion: "there" vs "their" (possessive)' },
  // "then" vs "than" — comparative misuse
  { re: /\b(more|less|better|worse|higher|lower|greater|smaller|larger|faster|slower)\s+then\b/i,
    msg: 'Possible confusion: "then" (time) vs "than" (comparison)' },
  // "effect" vs "affect" as verbs — "effect" as a verb means "to bring about" (rare/formal)
  // Only flag in clearly casual contexts
  { re: /\bthis\s+(?:will|can|may|might|would|could)\s+effect\s+(?:the|our|your|their|his|her|my|a|an)\b/i,
    msg: 'Possible confusion: "effect" (noun/formal verb "to bring about") vs "affect" (verb "to influence")' },
]
```

**Estimated QWK gain:** 0.03–0.05  
**False positive risk:** LOW — each pattern requires specific context beyond the homophones alone  
**Note:** "then/than" comparison is the highest-confidence pattern (~0% FP on native academic text)

## Proposal 3: Discourse Connector Coherence Check (organization.js) — MEDIUM VALUE

### Research Basis

Halliday & Hasan (1976) "Cohesion in English": discourse connectors only create cohesion when their semantic relationship matches the local discourse context. Using "however" without a prior opposing claim, or "therefore" without a prior cause, signals rote memorization rather than genuine command of discourse relations.

Granger & Tyson (1996) "Connector Usage in the English Essay Writing of Native and Non-native EFL Speakers": Chinese L1 writers overuse "however" as a generic paragraph opener (38% of uses not following a contrast), vs native writers (7% misuse).

**Conservative implementation — only the clearest misuse patterns:**

```js
// organization.js — connector coherence (basic semantic constraint check)
// Granger & Tyson 1996: Chinese L1 misuses "however" 38% as generic opener
// Only flag when "however" starts a paragraph with no preceding contrast-creating content
function detectConnectorMisuse(text) {
  const paragraphs = text.split(/\n\n+/).filter(p => p.trim().length > 20)
  let misuses = 0
  
  for (let i = 1; i < paragraphs.length; i++) {
    const para = paragraphs[i].trim()
    const prevPara = paragraphs[i-1].trim()
    
    // "However" as first word of a paragraph that follows a paragraph ending in agreement
    const startsWithHowever = /^however[,\s]/i.test(para)
    if (!startsWithHowever) continue
    
    // If preceding paragraph has no contrast-able content (ends with affirmative, no caveats)
    const prevHasContrast = /\b(but|yet|although|though|despite|while|whereas|however|nevertheless)\b/i.test(prevPara)
    if (!prevHasContrast) misuses++
  }
  return misuses
}
// Penalty: each misuse → -0.03 on organization score
```

**Estimated QWK gain:** 0.01–0.03  
**False positive risk:** MEDIUM — requires careful threshold tuning  
**Recommendation:** Implement conservatively (only paragraph-initial "however" after clearly non-contrastive paragraph). Limit penalty to max 1 misuse (cap at -0.03).

## Proposal 4: Email Register Consistency (relevance.js or style.js) — LOW-MEDIUM VALUE

### Research Basis

ETS TOEFL Writing Rubric (2023 update, Task 1 Email): "appropriate register and tone for professional/academic communication." Chinese L1 writers frequently mix informal and formal register within emails (research by Chen 2019, "L2 Email Pragmatics").

Current gap: style.js penalizes casual terms but does NOT reward consistent formal register in email context. Emails with fully formal vocabulary should score slightly higher on the style dimension.

**Implementation:**

```js
// style.js — email register consistency bonus
// Chen 2019: formal register consistency is Score-5 differentiator in email tasks
// Only triggers for 'email' taskType
function emailRegisterBonus(text, taskType) {
  if (taskType !== 'email') return 0
  
  // Formal email openers (beyond basic salutation)
  const hasFormalOpener = /\b(I am writing to|I would like to|I am pleased to|I hope this|Please find|With regard to|Regarding)\b/i.test(text)
  // Formal email closings
  const hasFormalClosing = /\b(I look forward to|please do not hesitate|Thank you for your|I would appreciate|I await your|Best regards|Sincerely|Yours faithfully)\b/i.test(text)
  
  return (hasFormalOpener && hasFormalClosing) ? 0.04 : (hasFormalOpener || hasFormalClosing) ? 0.02 : 0
}
```

**Estimated QWK gain:** 0.01–0.02 (email task only)  
**False positive risk:** VERY LOW  
**Recommendation:** Implement as small bonus. Many email practice essays already contain these patterns.
