## Pluralization Gap (Newly Identified)

The current template approach `t(key, { n: count })` works for Chinese (no plurals) but is awkward for English plurals:

- `"card{s} due tomorrow"` — English needs `card` vs `cards`
- `"Exam Mode · {days} day{s}"` — `day` vs `days`  
- `"knowledge point{s}"` — point vs points

**Current workaround in codebase**: inline ternaries like `day${examDays > 1 ? "s" : ""}`. These are not migratable to the current `t()` without adding plural support.

**Options:**
1. **Accept hybrid**: keep English plurals as inline JS, wrap only the outer string in `t()`. E.g. `t("review.examMode", { days: examDays, s: examDays > 1 ? "s" : "" })`
2. **Add plural key variants**: `"review.examModeSingular"` / `"review.examModePlural"` — verbose but typesafe
3. **Adopt ICU MessageFormat**: proper pluralization via `{count, plural, one{day} other{days}}` — needs a parser, overkill at current scale

**Recommendation**: Option 1 (pass `s` as a variable). Zero new infrastructure. Consistent with existing `{n}` pattern. Accept the mild awkwardness.
