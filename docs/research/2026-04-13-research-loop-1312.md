# Research Loop 13:12 — Claude.app Helper Process Leak 治本方案

## 问题概述

macOS 的 Claude Desktop（官方应用，非浏览器）在运行 MCP 子会话时，会 fork 出独立的 `claude` helper 二进制（`~/Library/Application Support/Claude/claude-code/<ver>/claude.app/Contents/MacOS/claude`），退出后 helper 不被回收，长时间驻留占用内存。本地实测当前版本 2.1.92 已累计 168 个残留 helper、约 2.5 GB RSS，最老 22 小时。上游已确认这是已知 bug（macOS 26.2 Tahoe 上尤其严重），但官方尚未完全修复，需要本地治本方案。

## 1. 上游已知状态

- **Issue #23496** `[BUG] Critical macOS 26.2 Memory Leak — Claude Desktop Background Processes`（2026-02-05 报告）：明确描述「背景 helper 保留巨量 heap，强 kill 后立即恢复」，与当前现象完全一致。([链接](https://github.com/anthropics/claude-code/issues/23496))
- **Issue #19201** `[BUG] Claude Desktop macOS leaves orphaned ...`：Desktop 关闭后 3 天还能查到孤儿 claude 进程。([链接](https://github.com/anthropics/claude-code/issues/19201))
- **Issue #16508 / #18859 / #11377 / #4953**：同类 memory leak，最大一单报 120 GB RAM 被 OOM 杀掉。
- **社区 workaround**：`Cresnova/claude-desktop-mcp-fix` 提供 wrapper 脚本防止 MCP 重复 spawn（基于 v0.12.129 测试）。([链接](https://github.com/Cresnova/claude-desktop-mcp-fix))
- **官方 changelog** 只修复了「Remote Control permission handler 保留」一处泄漏；helper fork 泄漏无 fix 版本号承诺。
- **结论**：上游已知，无 upstream fix ETA。必须本地兜底。

## 2. 治本方案 A：LaunchAgent 自动清理

### plist（`~/Library/LaunchAgents/com.user.claude-helper-cleanup.plist`）

```xml
<?xml version="1.0" encoding="UTF-8"?>
<!DOCTYPE plist PUBLIC "-//Apple//DTD PLIST 1.0//EN" "http://www.apple.com/DTDs/PropertyList-1.0.dtd">
<plist version="1.0">
<dict>
  <key>Label</key><string>com.user.claude-helper-cleanup</string>
  <key>ProgramArguments</key>
  <array>
    <string>/bin/bash</string>
    <string>/Users/shaoq/.local/bin/claude-helper-cleanup.sh</string>
  </array>
  <key>StartInterval</key><integer>3600</integer>
  <key>RunAtLoad</key><true/>
  <key>StandardOutPath</key><string>/tmp/claude-helper-cleanup.log</string>
  <key>StandardErrorPath</key><string>/tmp/claude-helper-cleanup.err</string>
</dict>
</plist>
```

加载：`launchctl load ~/Library/LaunchAgents/com.user.claude-helper-cleanup.plist`

### 清理脚本（`~/.local/bin/claude-helper-cleanup.sh`）

```bash
#!/bin/bash
# 杀掉运行超过 30 分钟的 claude helper（Claude.app 内部 fork 出的独立进程）
# 放过主 Claude.app 进程本体（父进程是 launchd 或 Finder）
THRESHOLD_SEC=1800
ps -axo pid=,etimes=,ppid=,command= | \
  awk -v t="$THRESHOLD_SEC" '
    $2 > t && $0 ~ /Application Support\/Claude\/claude-code\/[0-9.]+\/claude\.app\/Contents\/MacOS\/claude/ {print $1}
  ' | while read pid; do
    # 先 TERM 再 KILL，给它 3 秒清理锁文件
    kill -TERM "$pid" 2>/dev/null
    sleep 3
    kill -0 "$pid" 2>/dev/null && kill -KILL "$pid"
    echo "$(date '+%F %T') killed helper pid=$pid"
  done
```

### 注意事项

- 用 `etimes`（秒）比 `etime`（格式化字符串）好解析。
- 关键是 **命令路径正则精确匹配 helper 二进制路径**，避免误杀主 app 或 `claude` CLI。
- 首选 `StartInterval` 而非 cron：launchd 能保证机器休眠唤醒后补跑；cron 在 GUI session 下权限不全。
- 不用 `KeepAlive.OtherJobEnabled`——那是在别的 job 启动时联动，不适合定时清理。

## 3. 治本方案 B：进程监控工具对比

| 工具 | 自动告警 | 批量清理 | 成本 | 推荐度 |
|------|---------|---------|------|-------|
| btop | 否 | 交互式 kill | 免费 OSS | 日常看 ★★★★ |
| NeoHtop (Tauri) | 否 | 交互式 kill | 免费 OSS | 现代 UI ★★★ |
| Stats (exelban) | 菜单栏阈值提醒 | 否 | 免费 OSS | 常驻 ★★★★ |
| iStat Menus | 阈值提醒 | 否 | 付费 ~$12 | 商业党 ★★★ |
| LaunchAgent + 脚本（方案 A） | log 文件 | **是** | 零 | **★★★★★** |

**结论**：市面 GUI 工具都是「看」而非「自动清理」。要自动 kill，只有 LaunchAgent + 脚本路线。
