## 风险 1：DashScope 端点区域与 API Key 不匹配

### 现状

`src/lib/ai.ts` 第 9-11 行：

```ts
const qwen = createOpenAI({
  baseURL: "https://dashscope.aliyuncs.com/compatible-mode/v1", // 中国大陆
  apiKey: process.env.DASHSCOPE_API_KEY ?? "",
});
```

### 关键发现

DashScope 有三个独立的区域端点，**API Key 跨区域不可互用**：

| 区域 | 端点 | API Key 来源 |
|------|------|-------------|
| 中国大陆（北京）| `dashscope.aliyuncs.com` | 阿里云（国内）控制台 |
| 国际（新加坡）| `dashscope-intl.aliyuncs.com` | Alibaba Cloud（国际）控制台 |
| 美国（弗吉尼亚）| `dashscope-us.aliyuncs.com` | Alibaba Cloud（国际）控制台 |

**中国大陆 API Key 不能用于 `dashscope-intl` 端点，反之亦然。**

### 部署到 Vercel 时的风险

- Vercel 函数（即使在 hnd1/日本）从海外调中国大陆端点，请求走国际公网进入中国
- 海外 → 大陆的 HTTPS 可能受到 GFW 干扰（间歇性丢包/连接重置）
- 即使能通，延迟也比大陆内要高（50-200ms 额外开销）

### 判断方法

用户需要检查：
1. `DASHSCOPE_API_KEY` 是从哪个控制台获取的？
   - [国内控制台](https://dashscope.console.aliyun.com/)（登录用阿里云账号）→ 只能用 `dashscope.aliyuncs.com`
   - [国际控制台](https://bailian.console.alibabacloud.com/)（登录用 Alibaba Cloud 账号）→ 可用 `dashscope-intl.aliyuncs.com`
2. 本地开发测试时是否能正常调用？（能 = Key 和端点匹配；Vercel 上失败可能是网络问题）

### 建议

**若 Key 是国际账号**：立即把端点改为 `dashscope-intl.aliyuncs.com`（延迟更低、网络更稳定）。

**若 Key 是国内账号**：
- 短期：保持现状，接受偶尔网络抖动
- 长期：在 Alibaba Cloud（国际）注册，获取国际 Key，切换端点

## 风险 2：AbortSignal 全部注释导致 Vercel 函数静默挂起

### 现状

`src/lib/ai.ts` 中有 5 处 AI 调用，AbortSignal 全部被注释：

```ts
// abortSignal: AbortSignal.timeout(AI_TIMEOUT_MS), // TEMP DISABLED
```

注释原因：`// Temporarily disabled AbortSignal — investigating generate failures`

### 问题

Vercel 函数在 60s 时被硬杀死。**没有 AbortSignal 意味着**：
1. DashScope 网络挂起时，函数等 60s 才超时，全程消耗计费 CPU 时间
2. 客户端中途断开连接（如刷新页面），SSE 流式函数不会感知到，继续运行直到 60s 超时
3. Vercel 计费按实际执行时长，挂起函数导致浪费费用

### 正确修复

**问题根因（推测）**：调查 generate failures 时发现 AbortSignal 可能早于 DashScope 响应就触发。原因大概率是：DashScope 在大流量时首 token 延迟 > 55s，AbortSignal 提前取消了本该成功的请求。

**正确方案：延长超时，不是删除 AbortSignal**

```ts
// 原来：55s（可能太短）
const AI_TIMEOUT_MS = 55_000;

// 建议改为：每个调用场景设置不同超时
// vision calls（PDF解析）慢 → 58s
// text calls（生成）→ 50s  
// 给 Vercel 留 2-10s 处理响应
```

具体修复：

```ts
// src/lib/ai.ts - 按调用类型分别设置超时
export async function parseSyllabus(...) {
  const { text } = await generateText({
    abortSignal: AbortSignal.timeout(58_000), // vision call，DashScope PDF解析慢
    model: visionModel,
    ...
  });
}

export async function generateLesson(...) {
  const { text } = await generateText({
    abortSignal: AbortSignal.timeout(50_000), // text call，较快
    model: textModel,
    ...
  });
}
```

### SSE 路由的额外问题

`generate-one/route.ts` 的 SSE streaming 不传 `request.signal`，客户端断开无法通知到 DashScope。

更好的写法（把 request.signal 透传给 AI 调用）：

```ts
// generate-one/route.ts 的 sseResponse handler 签名改为接收 signal
const chunkPromises = outline.map((_, index) =>
  generateSingleLessonChunk(course.title, outline, index, request.signal)  // 传入 signal
    .then(...)
);
```

然后在 `generateSingleLessonChunk` 中：

```ts
export async function generateSingleLessonChunk(
  courseTitle: string,
  outline: ChunkOutlineItem[],
  index: number,
  signal?: AbortSignal,  // 新增参数
): Promise<GeneratedChunk | null> {
  ...
  const { text } = await generateText({
    abortSignal: signal ?? AbortSignal.timeout(50_000),  // 优先使用外部 signal
    ...
  });
}
```
