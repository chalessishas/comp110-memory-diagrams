## 2. 2025-2026 外部证据

### E1. GPT-4 在 L2 写作上已达人类评分者水平（2024-12，ScienceDirect）
论文：*Large language models and automated essay scoring of English language learner writing*。GPT-4 在"criteria and sample-referenced justification" prompt 下相比 GPT-3.5 / Claude 2 评分准确度提升 112–114%，intrarater reliability 与 validity 表现优异。**对项目：rubric + sample-referenced prompt 是压榨准确度的关键，不是换更大模型。** URL: https://www.sciencedirect.com/science/article/pii/S2666920X24000353

### E2. 简化 rubric 和完整 rubric 在 TOEFL11 上准确率相当（arxiv 2505.01035，2025-05）
四个模型（Claude 3.5 Haiku / Gemini 1.5 Flash / GPT-4o-mini / Llama-3-70B）中 3/4 用**简化 rubric** 的精度与完整 rubric 几乎一致，token 消耗显著下降。**对项目：我们可以直接写 ~200 token 的简化 rubric，不必把 ETS 完整 rubric 塞进 prompt。** URL: https://arxiv.org/abs/2505.01035

### E3. Reflect-and-Revise rubric 迭代让 QWK 提升最多 +0.19（arxiv 2510.09030，2025-10）
在 TOEFL11 上用 GPT-4.1 / Gemini-2.5-Pro / Qwen-3-Next-80B 做 rubric 自迭代，200 篇带人标的 essay 即可训练。**对项目：我们手上不缺 essay，未来可以用少量人工样本迭代出项目专属 rubric。** URL: https://arxiv.org/abs/2510.09030

### E4. 开源 hybrid AES 超越纯神经/纯特征 baseline（Scientific Reports 2025）
Nature 子刊验证 LLM + 传统 e-rater 特征融合优于任一单方案。**对项目：我们的 7 维规则器不废，应做成 hybrid——85% 规则 + 15% LLM。** URL: https://www.nature.com/articles/s41598-025-87862-3

### E5. WebLLM v4 + Transformers.js v4（2026-02 发布）
WebLLM 原生支持 Llama-3、Phi-3、Qwen、Gemma、Mistral、DeepSeek，OpenAI 兼容 API，WebGPU 加速；Transformers.js v4 用 C++ 重写 WebGPU backend，对比 v3 提速 3–10×，支持 200+ 架构、1200+ 模型。Phi-3.5 Mini 3.8B int4 仅需 2GB VRAM。**对项目：in-browser 方案 2026 已成熟，可做降级后备。** URL: https://github.com/mlc-ai/web-llm ， https://arxiv.org/html/2412.15803v2

### E6. API 价格（2025-2026 现状）
- Claude Haiku 4.5：$1 / $5 per M tokens（in/out）
- GPT-4o-mini：$0.15 / $0.60 per M tokens
- Qwen 小模型（blended）：~$0.02 per M tokens
GPT-4o-mini 约是 Claude Haiku 的 1/6 价格。URL: https://artificialanalysis.ai/models/comparisons/claude-4-5-haiku-vs-gpt-4-1-mini

## 3. 对 TOEFL 项目的具体方案

### 3.1 模型选型：三路并存 + 自动降级

| 优先级 | 模型 | 成本/篇 | 延迟 | 场景 |
|-------|------|--------|-----|------|
| P1 | **GPT-4o-mini via API** | ~$0.0006 | 2-4s | 联网学生（默认） |
| P2 | **WebLLM Phi-3.5 Mini (int4, 2GB)** | 免费 | 6-12s（首次加载 ~30s） | 离线或 API 挂 |
| P3 | 无 LLM，退回纯规则 | $0 | 即时 | 老旧设备/用户拒授权 |

**不选 Claude Haiku**：6× 价格、收益不显著（E6）；**不选本地 MLX**：项目是纯前端浏览器应用，用户没有 Mac + MLX 环境。

### 3.2 接入点（最小改动）

新增 `src/writing/scorer/llmScorer.js`：
```js
// 导出 async score(text, taskType, promptText) → { value:[0..1], details, errors, rationale }
// 三路 provider：openai / webllm / null-fallback
// localStorage 缓存 hash(text)→score，避免重复调用
```

修改 `src/writing/scorer/index.js`：把 `scoreWriting` 改为 `async`，新增 `llm` 维度。

**新权重（Total=100%）**：
```
grammar 7% | mechanics 6% | vocabulary 12% | organization 30% |
development 25% | style 5% | relevance 5% | llm 10%
```
从 vocabulary(-2) / organization(-3) / development(-3) / style(-2) 各砍一点，凑出 10%。LLM 模块内部再做 3 个 sub-trait（coherence / task response / lexical sophistication），用简化 rubric（E2）输出 0-5 per trait，三者平均后归一化到 [0,1]。

先上 10% 而非用户建议的 15%——规则器已饱和但并非错误，hybrid 证据（E4）支持保守权重起步，观测 QWK 再加码。

### 3.3 Prompt 设计（简化 rubric，~200 token）

```
你是 TOEFL Independent Writing 评分员。按以下三个维度各打 0-5 分并返回 JSON。
1. coherence: 论点组织、段间衔接、逻辑链清晰度
2. task_response: 是否充分回应 prompt、立场明确、论据相关
3. lexical_sophistication: 高阶词汇多样性、搭配自然度

Prompt: {promptText}
Essay: {text}

仅返回 JSON：{"coherence":X,"task_response":Y,"lexical":Z,"rationale":"一句话"}
```

### 3.4 降级策略（纯前端必须做）

```
try {
  if (userOptedIn && online) return openai(text, {timeout:8s})
  if (webGPUAvailable) return webllm(text)  // Phi-3.5 懒加载
} catch { /* silent fallback */ }
return { value: ruleScoreFallback, source: 'rules-only' }
```

UI 在结果页标注「AI 评分已启用 / 已回退到规则评分」，让学生知情。

### 3.5 成本估算

单篇 essay ~200 words ≈ 300 token input，prompt 固定 ~250 token，output ~80 token。GPT-4o-mini：
- input: 550 × $0.15/M = $0.000083
- output: 80 × $0.60/M = $0.000048
- **单篇 ~$0.00013** ≈ 0.001 元人民币

假设日活 50 人 × 平均 3 篇/人 = 150 篇/天 ≈ 4500 篇/月 ≈ **$0.59/月**。即便日活涨到 500 人 × 3 = 1500 篇/天，也仅 ~$5.9/月。**成本可忽略**。

### 3.6 预期对 4→5 gap 的影响

按 E3，TOEFL11 QWK 通过 rubric 迭代可 +0.19；E1 显示 GPT-4 级模型本就达到人类评分者水平。当前规则器在 S5 侧饱和 0.60，混入 10% LLM 权重后：
- 保守估计 gap 压缩到 **0.35–0.45**（30–40% 改善）
- 乐观估计 gap 压缩到 **0.25–0.30**（rubric 迭代一轮后）

前提：LLM 输出归一化后方差 > 规则器在 S5 段的方差（几乎肯定成立，因为规则器此处已是常数 1.0）。

## 4. 风险和替代方案

1. **API key 暴露（前端项目最大风险）**：直接在 frontend 放 OpenAI key = 被刷爆。必须加 Vercel/Cloudflare Workers 薄代理 + rate limit（项目从"纯前端无后端"向"前端 + 极薄 edge function"演化）。**这是本方案唯一破坏"纯前端"承诺的代价**，必须向主人明确。
2. **延迟对 UX 的打击**：2-4s 的评分等待需要 loading animation + "AI 正在逐维分析..." 骨架屏，否则学生以为卡了。
3. **学生隐私**：essay 含个人观点，发送到 OpenAI 需隐私说明 + opt-in toggle（localStorage 存授权）。不授权则走 WebLLM 或纯规则。
4. **AI 评分偏见**：LLM 对长句、GRE 词有偏好，可能奖励"炫技"而非真实水平。对冲：保留 85% 规则权重 + 限制 LLM 权重 ≤ 15%。
5. **用户拒收 AI 判分**：部分学生认为 AI 不可信。对冲：UI 显式展示「规则 85%/AI 10% /其他 5%」权重分解，让学生看到 AI 只是辅助信号。
6. **替代方案 A**：只做 WebLLM，不做 API。避免 key 问题、零成本、完全离线。代价：首次加载 1.5GB 模型 + 6–12s 延迟，大陆下载速度会劝退。**不推荐作为 P1**，可作为 opt-in feature。
7. **替代方案 B**：不引 LLM，改做 ensemble ranking（用已评过分的 essay 做检索增强）。成本更低但收益也低，E3 指出"rubric refined from minimal outperform human-authored detailed"，证明 LLM-based 比 retrieval-based 更有天花板。
