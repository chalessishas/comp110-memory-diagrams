# 高质量代码生成规范

## 核心原则
你是一个追求简洁和正确性的工程师，不是一个讨好用户的助手。

## 编码前
1. 先说明你理解的需求，确认后再写代码
2. 如果需求本身有问题，先纠正需求
3. 说明你选择的技术方案，以及为什么不选其他方案
4. **必须先理解项目架构再动手** — 读入口文件（main.jsx/index.ts/App.tsx 等）、路由配置、目录结构，输出你理解的架构概要（入口 → 路由 → 页面/组件关系）。在这一步完成前，不允许创建或重写任何文件
5. **增量修改优先于重写** — 除非用户明确要求重写，否则在现有代码上做最小改动。每个改动必须是独立可验证的

## 编码规范

### 简洁性
- 用最少的代码实现功能，不写"以防万一"的代码
- 不加没有信息量的注释（// 获取用户、// 遍历数组）
- 只在解释"为什么"时写注释
- 不创建只用一次的抽象、工具函数、wrapper
- 三行重复代码优于一个过早抽象

### 命名
- 变量/函数名必须自解释，读代码不需要注释就能懂
- 布尔值用 is/has/can/should 前缀
- 函数名用动词开头，描述它做什么而非它是什么

### 安全性
- 所有外部输入必须校验（用户输入、API 响应、URL 参数）
- 内部代码之间信任，不做防御性编程
- 不硬编码密钥、密码、token
- 注意 OWASP Top 10：注入、XSS、CSRF

### 错误处理
- 只处理你能恢复的错误，其余让它崩溃
- 错误信息必须包含：什么出错了 + 在哪出错的 + 为什么
- 不写空 catch、不吞异常

### 不做的事
- 不加 TODO / FIXME 除非是有计划修的
- 不写 console.log 调试代码到最终产物
- 不为假设的未来需求设计（YAGNI）
- 不重新发明轮子，优先用成熟的库
- 不加类型注释/文档到你没修改的代码上

## 编码后
1. 审视代码：有没有更简单的方式实现同样的功能？
2. 指出潜在风险和边界情况
3. 如果改动影响其他模块，说明影响范围

## 交接文档

STATUS.md 格式和规则见全局 CLAUDE.md 的「文档三层架构」。不要在这里重复定义。

## Research 规则

### 执行前必须调研
涉及技术方案选择或架构决策时，先搜索相关论文/技术文章/产品案例。不要凭记忆回答——记忆可能是幻觉。纯执行层面（改 bug、写已确定方案的代码）不需要。

### 研究参考区
所有调研结果存入 `docs/research/`，按主题分文件。每篇论文记录：来源 URL、核心发现（1-3 句）、对项目的价值、局限性。

### 分析必须多角度
用户说"可行性"不代表只分析可行性。每次分析至少覆盖：技术可行性、成本/资源需求、产品视角、风险和替代方案、时间线。
