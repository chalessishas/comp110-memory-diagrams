# Research Loop — 2026-04-21

## 1. Pytest Parametric Test Generation (300+ operator files)

**Current pain:** 376 test files, each with its own `_world()` / `_ally()` / `_enemy()` helpers — massive duplication.

**Best practices found:**

- **Central conftest fixtures** — move `_world()`, `_ally()`, `_enemy()` into `tests/conftest.py` as named fixtures; all 376 files import nothing, they just declare fixture arguments
- **`pytest_generate_tests` hook** — call `metafunc.parametrize()` inside this hook in conftest to drive data-table-driven tests dynamically; avoids per-operator `@parametrize` boilerplate entirely
- **Indirect parametrization** — `@pytest.mark.parametrize("op_factory", [...], indirect=True)` lets the fixture do expensive world setup once per operator; test body stays 3–5 lines
- **Operator test table pattern** — define one dict per operator (`{"name": "aak", "skill": 2, "sp_cost": 30, "duration": 15, ...}`), store in `tests/data/operator_skills.py`, run a single `test_skill_config[aak-s2]` parametrized test instead of one file per operator
- **Factory fixture** — a `make_world` fixture factory (returns a callable) lets individual tests configure scenario without duplicating setup code
- **`pytest-subtests`** — for grouping many assertions per operator without separate test functions; keeps failure reporting granular

**Concrete next step:** Consolidate skill-config tests (SP cost, duration, slot, name) into one parametrized file covering all operators. Keep scenario tests (buff application, damage values) per-operator but share fixtures via conftest.

## 2. ECS Combat Simulation Patterns (real-time/tick hybrid)

- **Deterministic tick loop** — all systems run in fixed declaration order each `engine.tick()`; critical for replay and anti-cheat (Blizzard Overwatch 2 model at 63Hz)
- **Phase scheduling** — enforce order: `input → SP gain → skill activation → damage application → buff decay → death/respawn`; avoids causality bugs
- **Event queue for sparse events** — skill triggers and damage events go through a queue rather than every system scanning all units every tick
- **Buff/debuff as duration components** — store `remaining_ticks` on the buff entity; a single `BuffDecaySystem` handles all expirations uniformly
- **Batch stat writes** — accumulate ATK/DEF modifiers in a staging component, apply once per tick in a `StatFlushSystem`; prevents mid-tick inconsistencies
- **PendingDestroy tag** — never delete entities mid-loop; tag for cleanup, run `DestroySystem` last in the phase
- **Archetype churn watch** — skill activation that adds/removes many components causes table migrations; isolate volatile components (active skill state) into separate tables

**Relevance to arknights-sim:** Current `register_default_systems` already has ordered phases — the gap is likely buff decay and stat flush not being batched, causing test sensitivity to tick ordering.

## 3. Arknights Skill Data Sources Beyond Kengxxiao

| Source | Format | Notes |
|--------|--------|-------|
| `Kengxxiao/ArknightsGameData` | Raw JSON (CN) | Primary source; `skill_table.json` has SP costs, duration, levels |
| `Kengxxiao/ArknightsGameData_YoStar` | Raw JSON (EN/JP/KR) | EN localization; same schema |
| `ArknightsAssets/ArknightsGamedata` | Lua → JSON via arkprts | Auto-fetched; covers CN/EN/JP/KR/TW/Bili; good for diff tracking |
| `HellaOrg/HellaAPI` | REST API (MongoDB) | Parsed operator data; useful for quick queries without local clone |
| `wesngu28/rhodesapi` | REST API (Railway) | Scraped from Gamepress; less reliable for numeric values |
| `aceship.github.io` | Web toolbox | Human-readable; not suitable for automated ingestion |
| `arkntools/arknights-toolbox` | JS computed | Has preprocessed talent/skill data structures |

**Recommendation:** `ArknightsAssets/ArknightsGamedata` is the best complement to Kengxxiao — automated CI updates, multi-region, same JSON schema. Use it to catch CN-ahead data before EN releases. For CI validation of skill parameters (SP cost, duration), parse `skill_table.json` directly and assert against test constants.

**Quick win:** Write a `tools/validate_skill_constants.py` that loads `skill_table.json` and cross-checks every `_SP_COST`, `_DURATION` constant in `data/characters/*.py` against game data — catches drift automatically.
