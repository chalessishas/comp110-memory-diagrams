## Topic 1: W Operator S3 "Chaos Edict" — Bomb Delay Mechanics

### Confirmed Mechanics

**Skill name at M3:** D12 (renamed at mastery; also called "Chaos Edict" in some translations)

**Bomb delay:** 3 seconds after skill activation / bomb placement  
Source: arknights.wiki.gg/wiki/W (confirmed via two fetch passes) — "immediately plant an explosive charge on multiple enemies within her range which detonates after 3 seconds"

**Timer start:** The 3-second countdown begins at the moment of skill activation (simultaneous with charge placement on targets). The wiki does not distinguish between "activation time" and "placement time" — they are treated as the same moment.

**If target leaves the map before detonation:** Premature detonation at the target's last position. Does NOT cancel — the bomb explodes regardless. Enemies that merely move within the map do NOT trigger premature detonation (only full map-exit does).

**Stats at M3 (max rank):**
- Targets: up to 4 enemies, prioritized by highest current HP
- Damage: 310% ATK as Physical Damage per bomb (AoE)
- Stun duration on damaged targets: 5 seconds
- AoE radius per bomb: 1.2 tiles
- ATK snapshot: uses W's ATK value at the moment of skill activation (not at detonation)

**SP cost / initial SP:** sp_cost=50, initial_sp=20, AUTO_TIME, AUTO trigger

All three constants are wrong relative to wiki values. The implementation architecture (EventQueue, snapshot ATK at activation, position payload) is correct.


**Estimated LOC:** 3 (pure constant fixes, no logic change)  
**Risk:** Tests that assert specific damage values will need expected values updated. The architecture is sound; only the calibration numbers are wrong.

**Multi-target behavior gap:** The current implementation hits all enemies in radius at the stored position in a single handler. The wiki says bombs are placed on up to 4 enemies individually, each bomb exploding at that enemy's position. If enemies spread out before detonation, each bomb should detonate at its own target's original position (not all at one position). Current code stores only one `(bx, by)` per S3 activation — this means one bomb at the target's position at cast time, not 4 individual bombs on 4 targets.

**Fix for multi-target:** In `_s3_on_start`, iterate over current targets (up to 4, by HP descending), schedule a separate `w_bomb_detonate_{uid}_{n}` event per target, each with its own `(bx, by)` at that target's current position. Estimated 20 additional LOC.

## Topic 2: Stun / Freeze vs SP Recovery

### Research Findings

The official wikis (arknights.wiki.gg and arknights.fandom.com) do NOT explicitly state whether Stun or Freeze blocks AUTO_TIME (time-based) SP recovery. After exhaustive searching across Stun, Freeze, Negative Status, Skill Point, and Status Resistance pages, none of them contain a direct statement like "Stun blocks automatic SP gain."

**What the wikis do confirm about Stun:**
> "Unable to move, block, attack, or use skills."

The four listed disabilities are: movement, blocking, attacking, skill usage. SP recovery is not in this list.

**Silence comparison:** Silence prevents skill usage but does not block SP recovery (SP continues accumulating while silenced; the operator just cannot activate the charged skill).

**Duelist Defender indirect case:** The Duelist Defender trait says "Only restores SP when blocking enemies." Since Stun prevents blocking, a Stunned Duelist Defender cannot recover SP — but this is an indirect effect of losing block, not a direct Stun → SP lockout rule. Other archetypes are not constrained this way.

**Absence of evidence:** No wiki page, community guide, or YouTube mechanic video found explicitly says "Stun blocks AUTO_TIME SP recovery for standard operators." The "Arknights Science: SP Lockout" video is referenced in search results but its content was not accessible via WebFetch.

### Best Available Interpretation

Community consensus (based on the prior research doc at `docs/research/2026-04-19-2330-next-priorities.md`) claimed Stun blocks AUTO_TIME SP. This claim is NOT verified by primary wiki sources in this research pass. It appears to be a community assumption that propagated between guides without being independently confirmed.

**Two plausible real-game behaviors:**
1. **Stun does NOT block AUTO_TIME SP** — consistent with the wiki listing only 4 disabled functions (move, block, attack, skills). SP would still tick up during stun; the operator just can't activate the charged skill.
2. **Stun DOES block AUTO_TIME SP** — consistent with the intuition that a fully disabled unit shouldn't be "mentally preparing their skill," and consistent with the previous research claim.

**Most likely correct behavior based on game logic:** Behavior 1 (Stun does NOT block AUTO_TIME SP). The wiki is exhaustive about what Stun disables and SP recovery is absent. Silence is the status that specifically targets skill-related functions, and even Silence does not block SP accumulation. The Duelist Defender case is the only documented SP-blocking behavior, and it's tied to the blocking mechanic, not stun directly.

### Current Simulator Status

`/Users/shaoq/Desktop/VScode Workspace/arknights-sim/core/systems/skill_system.py` line 39:

```python
if sk.sp_gain_mode == SPGainMode.AUTO_TIME and u.can_use_skill():
```

`can_use_skill()` (unit_state.py line 249) returns False when `can_act()` is False (i.e., during Stun/Bind/Freeze/Sleep/Levitate). This means the simulator currently **blocks AUTO_TIME SP during Stun** — matching interpretation 2.

If the real game uses interpretation 1 (Stun doesn't block SP), the fix is:

```python
# Change from:
if sk.sp_gain_mode == SPGainMode.AUTO_TIME and u.can_use_skill():
# Change to:
if sk.sp_gain_mode == SPGainMode.AUTO_TIME and u.alive and u.deployed:
    if sk.sp_gain_mode == SPGainMode.AUTO_TIME:
        # SP accumulates even while stunned; activation is gated elsewhere
```

Or more cleanly:
```python
if sk.sp_gain_mode == SPGainMode.AUTO_TIME:
    if now >= sk.sp_lockout_until:
        sk.sp = min(sk.sp + dt, float(sk.sp_cost))
```

(Remove the `can_use_skill()` check from the SP accumulation branch; keep it only on the trigger check.)

### Recommendation

**Status: UNVERIFIED — do not change the simulator without in-game testing.**

The current behavior (Stun blocks AUTO_TIME SP via `can_use_skill()`) is conservative and not clearly wrong. The practical gameplay impact is minor: Stun durations in-game are typically 1–5 seconds vs. skill SP costs of 20–50s, so the SP lost to stun is a small fraction of total charging time. Change priority is LOW until confirmed via direct game observation.

**Estimated LOC if fix needed:** 4 (remove `can_use_skill()` gate from SP branch; add separate check for skill activation). Risk is low since the change is surgical.

## Topic 3: ASPD Stat Mechanics — Formula and Frame Rounding

### Confirmed Formula

Source: arknights.wiki.gg/wiki/Attribute/Attack_interval (confirmed on two fetches)

```
a = (100 / (100 + x)) * (b + y)
```

Where:
- `a` = final attack interval in seconds
- `b` = base attack interval in seconds (from operator data, no modifiers)
- `x` = sum of all ASPD modifiers (additive — "+5 ASPD" and "+10 ASPD" = x=15)
- `y` = flat attack interval modifier from skills (seconds; only one flat modifier active at a time)

**Key constraints:**
- Base ASPD = 100 for all units
- ASPD clamped to [20, 600] — interval cannot exceed `b * 100/20 = 5× base` or go below `b * 100/600 ≈ 0.167× base`
- Only one flat interval modifier (`y`) can be active at a time; multiple ASPD modifiers stack additively
- Flat modifier applies BEFORE ASPD scaling (modifies effective base, not final result — note: the formula applies y as additive to b, then divides by ASPD scaling)

**Example:** Operator with b=1.6s, ASPD buff x=+50:
```
a = (100 / (100 + 50)) * 1.6 = (100/150) * 1.6 = 0.6667 * 1.6 = 1.067s
```

**Example with flat modifier:** Same operator, flat interval y=+0.4s:
```
a = (100/150) * (1.6 + 0.4) = (100/150) * 2.0 = 1.333s
```

### Frame Rounding

The official wiki and community documentation do NOT document the exact rounding method (floor, ceiling, or standard round) used to discretize the final interval to 30 FPS frames. The wiki acknowledges: "Since Arknights runs in 30 frames per second, the 0.285 sec interval is actually rounded (whether up or down remains unknown)."

Community testing has not resolved this definitively. The formula in the game is known; the frame-quantization step is not publicly documented.

**Practical implication:** For a 1.6s interval at 30 FPS = 48 exact frames. For 1.067s = 32.0 frames (exactly divisible). For 0.285s = 8.55 frames — this ambiguous case exists but is uncommon for standard operators.

### Current Simulator Status

`/Users/shaoq/Desktop/VScode Workspace/arknights-sim/core/state/unit_state.py` lines 234–237:

```python
@property
def current_atk_interval(self) -> float:
    aspd = max(20.0, min(600.0, self.effective_aspd))
    return self.atk_interval * 100.0 / aspd
```

This correctly implements `a = b * (100 / (100 + x))` since `effective_aspd` returns `100 + sum_of_buffs`. The ASPD clamping [20, 600] is implemented.

**Gap: flat attack interval modifier (`y`) is not implemented.** The formula includes `y` (a flat seconds offset to the base interval from skills), but `current_atk_interval` has no parameter for it. The `BuffAxis.ASPD` axis is used for ASPD percentage buffs but there is no `BuffAxis.ATK_INTERVAL` or equivalent for flat interval changes.

For most operators this doesn't matter — flat interval modifiers only appear on specific skills (e.g., Exusiai S3 reduces her interval by a flat amount). For those operators, ASPD-style buffs work; flat interval buffs are silently dropped.

**Gap: no frame rounding.** The interval is returned as a raw float. Since the simulator uses DT=1/30s ticks and compares `atk_cd <= 0`, sub-frame drift will accumulate over long fights. Prior research (2330 doc) noted this. The practical error per attack is ≤ 1/30s = 0.033s. Over a 180s fight with a 1.5s interval, that's 120 attacks × 0.033s max drift = 4s maximum accumulated error. In practice, random rounding of `atk_cd` means this averages out rather than accumulating.

### Implementation Recommendations

**Priority 1 (confirmed gap, minor):** Add `BuffAxis.ATK_INTERVAL` flat offset support in `current_atk_interval`:

```python
@property
def current_atk_interval(self) -> float:
    aspd = max(20.0, min(600.0, self.effective_aspd))
    flat_offset = self.effective_stat(BuffAxis.ATK_INTERVAL) if hasattr(BuffAxis, 'ATK_INTERVAL') else 0.0
    return (self.atk_interval + flat_offset) * 100.0 / aspd
```

Requires adding `ATK_INTERVAL = "atk_interval"` to `BuffAxis` enum in `core/types.py`.  
**Estimated LOC:** 8 (2 in types.py, 6 in unit_state.py)  
**Impact:** Required for correct Exusiai S3, Thorns S3, and other operators with flat interval modifiers

**Priority 2 (not urgent):** Frame rounding. Add `round(interval * 30) / 30` to `current_atk_interval`. Since exact rounding direction is unknown, use standard `round()` as the best approximation:

```python
# Optional: snap to 30fps grid
raw = (self.atk_interval + flat_offset) * 100.0 / aspd
return round(raw * 30) / 30
```

**Estimated LOC:** 1  
**Risk:** This could break existing tests that assert exact interval values. Test tolerance windows may need updating (+/- 1 frame tolerance).
